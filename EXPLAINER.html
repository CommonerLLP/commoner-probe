commoner·probe

Sousveillance infrastructure for the state's mandatory disclosure systems.

A commoner probes the state's own paperwork — parliamentary questions, committee reports, state-assembly records, and executive disclosures like corporate-CSR and mineral-fund spending — and turns it into clean, schema-validated evidence. This page explains what the project does, shows you the actual data it produces, and lets you assemble the commands that pull it.

Python ≥ 3.9 v0.4.0 MIT License JSONL output ● Layer 0 — acquisition only

01What it actually does

One job, done reliably: acquire Indian legislative and executive disclosure into clean JSONL. It deliberately does not classify, topic-model, or build dossiers — that is someone else's layer.

🏛️

Parliamentary questions

Starred & unstarred Q/A from Lok Sabha and Rajya Sabha — who asked, which ministry answered, the date, the session, and the PDF.

📋

Standing committee reports

Budget scrutiny, bill examinations, subject investigations, and Action Taken Reports — with the ATR linked back to the report it answers.

🗺️

State assembly records

Questions, members, and papers-to-be-laid from NeVA portals — the e-Vidhan platform that 28+ state legislatures are onboarding onto.

It also acquires executive disclosure — the spending the state reports about itself:

🏢

MCA CSR company spend

Corporate Social Responsibility CSV exports from the Ministry of Corporate Affairs CDM portal — company, financial year, sector, and amount spent.

⛏️

Mines DMFT / PMKKKY

Raw Ministry of Mines and Odisha DMFT disclosure files — District Mineral Foundation collections and project-fund snapshots, captured with hashes and provenance.

🔗

Cross-source evidence

Bundle executive disclosure (Mines DMFT) beside parliamentary oversight (Sansad Q/A) without flattening unlike source families into one table.

The output is the contract. Every probe writes append-only manifest.jsonl (one record per question/report), downloads the source PDFs, and logs a full audit trail in _runs.jsonl. Downstream tools consume that corpus — they never re-scrape.

02Why it has to exist

The data is mandatory disclosure — published under the Right to Information Act 2005, §4. It exists. It is just engineered to be practically inaccessible.

🚫 No documented API

The endpoints sansad.in's own JavaScript calls are undocumented, version without announcement, and differ structurally between Lok Sabha and Rajya Sabha.

📄 PDFs are the record

Web metadata is sparse. The substance lives in PDFs that must be downloaded and text-extracted to be read programmatically.

🌐 28+ bespoke portals

Each NeVA state assembly has its own subdomain and state code. None publish a central registry or endpoint docs.

📉 Unreliable uptime

Portals throttle and 5xx during sessions — exactly when the data matters most — with no Retry-After headers.

General scrapers (Scrapy, httpx, Playwright) give you HTTP but zero domain knowledge. They don't know that LS question search buckets by ministry while RS buckets by session number, that committee PDF paths are constructed from reportNo+uuid rather than linked, or that any of those fields need sanitising against path traversal before they touch your filesystem. commoner-probe is that domain knowledge, made executable.

A topic profile is the only thing you write. It tells the probe what to search for and which ministries to filter. This is the bundled libraries example — copy it, edit it, point a probe at it.

examples/topics/libraries.json
{
  "name": "libraries",
  "description": "Public library infrastructure and policy",
  "search_groups": {
    "public_libraries": ["public library", "rural library"],
    "policy": ["National Mission on Libraries", "RRRLF"]
  },
  "lok_sabha_ministries": ["CULTURE", "EDUCATION"],
  "rajya_sabha_ministry_likes": ["CULTURE", "EDUCATION"]
}
FieldWhat it controls
search_groupsKeyword groups for Lok Sabha full-text search. Each query runs independently; results are union-deduped on key.
lok_sabha_ministriesExact ministry filter for LS (case-sensitive).
rajya_sabha_ministry_likesLIKE / prefix match for RS — the two houses' APIs filter differently.

Five profiles ship built-in: libraries, affirmative_action, home_affairs_starred, mines_dmft_pmkkky, narcotics_substance. Drop one to disk with commoner-probe init-topic --name libraries --out topic.json.

Why two shapes? The two houses run different software. LS records carry uuid, handle, session, loksabhanumber, found_via_group. RS records carry qslno, ses_no, inline question_text/answer_text, and a Hindi PDF. Both share key, kind, house, date, qtype, ministry, askers. The library normalises what it can and preserves the rest.

After extract-answers parses the PDFs, each text unit becomes a record in answers.jsonl. Three shapes, discriminated by kind:

answers.jsonl · kind = "atr_response"
{
  "key": "LS|finance|41|18",            // joins back to the manifest record
  "kind": "atr_response",
  "recommendation_no": 7,
  "recommendation_text": "The Committee recommends that the Ministry expedite release of pending RRRLF grants to State library authorities...",
  "response_text": "The Government has noted the recommendation. Funds amounting to Rs. 25 crore have been released...",
  "confidence": 0.9,
  "extractor": "answers_regex_v1",
  "source_pdf": "pdfs/ls/18_Finance_41.pdf"

kindFrom which PDFCarries
qa_responseQ/A PDFsquestion_text + answer_text pairs, split at a boundary marker
atr_responseAction Taken Report PDFs(recommendation_no, recommendation_text, response_text) triples
dfg_recommendationDFG / bill / subject PDFsnumbered observation paragraphs

Each record keeps a confidence score and the extractor version, so you can filter on parse quality and re-run as the extractor improves.

📄 extract-answers — parse PDFs → text▸

Reads manifest.jsonl + downloaded PDFs and writes answers.jsonl with Q/A pairs, ATR recommendation/response triples, and DFG observation paragraphs. Requires pip install "commoner-probe[pdf]".

example
commoner-probe extract-answers --out data/climate
commoner-probe extract-answers --out data/climate --refresh # force re-parse
🔗 atr-linkage — connect ATR → original report▸

For every Action Taken Report, parses the title to find the original committee report it responds to, writing atr_linkage.jsonl. Idempotent — safe to re-run.

example
commoner-probe atr-linkage --out data/committees

This is what enables lifecycle analysis: recommendation → government acceptance/rejection → follow-up.

08How the files join

The keys are designed to cross-reference. This is what makes the corpus a graph, not just a pile of records.

manifest.jsonl .key ←→ answers.jsonl .key manifest.jsonl .run_id ←→ _runs.jsonl .run_id manifest.jsonl .key ←→ atr_linkage.jsonl .atr_key (ATR side) atr_linkage .references_report_key ←→ manifest.jsonl .key (original side) manifest.jsonl .asker_entity_ids[] ←→ entities/people.jsonl .entity_id

The ATR life-cycle chain — the research-grade unit of analysis

This is the payoff: trace a committee's recommendation all the way to the government's written response.

manifest (committee_report · report_type = demands_for_grants | bill | subject) └─ answers (kind=dfg_recommendation) ← the committee's original recommendation └─ atr_linkage.references_report_key └─ manifest (committee_report · report_type = action_taken) └─ answers (kind=atr_response) ← the government's response

Design principles that fall out of this

Boundaries are files

Every layer hands the next a JSONL file on disk. You can inspect, validate, and re-run any stage independently — no hidden in-memory state.

Provenance is preserved

Session, question number, ministry, date, topic hash, and per-bucket counts travel with the data. The apparatus of acquisition is as auditable as the data it acquires.

Idempotent & resumable

Records dedup on key; crawls resume from what's already in the manifest. Re-running is cheap and safe.

Acquisition only

No classification, no search index, no normative judgement. A sharp, testable scope keeps the contract — clean JSONL — stable for everything downstream.

Why "Layer 0"? commoner-probe is the acquisition floor of the CommonerLLP toolchain (imported as import commoner_probe as probe). Topic classification and semantic search live in a compose layer above it that injects a filter_fn into the topic profile — out of scope here by design.

10Loading a corpus in Python

Once the JSONL exists, probe.Corpus gives you typed iterators and join helpers.

quickstart.py
import commoner_probe as probe
from collections import Counter

c = probe.Corpus("data/climate")

# Which ministries field the most questions on this topic?
counts = Counter(r.ministry for r in c.manifest_qa())
for ministry, n in counts.most_common(10):
    print(f"{ministry}: {n}")

# Join manifest records to their extracted answer text
for pair in c.join_qa():
    if pair.answers:
        print(pair.manifest.title)
        print(pair.answers[0].question_text[:200])

# Walk the ATR life-cycle: recommendation → government response
for chain in c.join_atr_chain():
    print(f"Report: {chain.original and chain.original.title}")
    print(f"  recommendations: {len(chain.original_observations)}")
    print(f"  govt responses:  {len(chain.atr_answers)}")

# Straight to pandas (pip install commoner-probe[pandas])
df = c.to_dataframe("manifest_committee_reports")
